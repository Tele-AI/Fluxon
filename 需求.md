# 远端压测 testbed 需求

## 结论

GitHub Actions 不应只依赖 GitHub 提供的免费单机资源。压测主路径需要切到仓库外部的 shared Fluxon testbed：由 GitHub Actions 通过 SSH 触发 `44`、`46` 机器上的 testbed，再由远端 testrunner 执行压测。

## 目标

- 在 `44`、`46` 机器上长期维护可复用、可恢复的 shared Fluxon testbed。
- GitHub Actions 只负责触发、参数下发、结果回收，不承担实际压测资源。
- 远端执行入口继续沿用 teststack 的两步契约：
  1. `start_test_bed.py` 负责启动或复用 shared testbed。
  2. `test_runner.py` 负责执行压测 suite / benchmark。
- 后续可以扩展到更大规模集群，但外部触发模式保持一致：`GitHub Actions -> SSH trigger -> remote shared testbed -> remote testrunner`。

## 稳定契约

- GitHub Actions 侧需要一个明确的远端目标选择面，建议使用显式 `testbed_cluster_id`，首个值为 `testbed_44_46`。
- `44`、`46` 机器上的 testbed 必须是稳定的长生命周期环境，不能假设每次 CI 都从零人工搭建。
- UI authority 仍归 testrunner；如果需要 UI，应作为远端长生命周期服务运行，并复用底层 ops 接口。
- 不把入口设计成开放式“任意 SSH 主机 + 任意参数”模式；后续新增集群时，增加新的显式 `testbed_cluster_id` 即可。

## 待细化

- GitHub Actions 如何选择目标集群、场景和压测规模。
- SSH 认证、远端工作目录、release / test_rsc 同步方式。
- 远端 testbed 的并发调度、互斥、失败恢复和日志回收策略。
